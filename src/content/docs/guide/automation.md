## 概覽

Automation Platform 是 Arova Nexus 的流程自動化引擎，目標是把「人手動做的重複工作」變成「系統自動完成的流程」。

核心概念是**工作流程（Workflow）**：你用視覺化編輯器把一系列動作串成一條流程，設定觸發條件後，系統會按照你設計的順序自動執行每個步驟。

一個典型的工作流程長這樣：

```
觸發器 → 條件判斷 → HTTP 呼叫 AD → 人工簽核 → 通知使用者
```

你可以用 Automation Platform 做到：

- **設計流程**：拖拉式視覺化編輯器，不需要寫程式就能組合流程
- **排程執行**：設定 Cron 讓流程定時自動跑
- **外部觸發**：讓其他系統透過 Webhook 或 API 啟動流程
- **即時監控**：圖形化追蹤每個節點的執行狀態
- **版本控制**：每次發布建立新版本，可隨時回滾到舊版

## 瀏覽工作流程

**適用角色：** IT 管理員、IT 維運人員

進入 Automation Platform 後，你會看到所有已建立的工作流程清單。

### 操作步驟

1. 點選左側選單的**自動化**
2. 你會看到所有流程定義的清單頁面，每筆顯示：
   - 流程名稱
   - 說明
   - 最近一次執行狀態（成功 / 失敗 / 未曾執行）
   - 目前版本號
3. 點選右上角的檢視切換按鈕，在**卡片式**和**列表式**之間切換
   - **卡片式**：適合快速瀏覽，一眼看到執行狀態
   - **列表式**：適合大量流程的管理，資訊更密集
4. 使用頁面上方的**搜尋列**輸入關鍵字，快速找到特定流程
5. 使用**篩選**功能縮小範圍（例如只看失敗的流程、特定分類的流程）
6. 點選**排序**下拉選單，可依「最近執行」或「名稱」排序

## 視覺化流程編輯器

**適用角色：** IT 管理員

流程編輯器是 Automation Platform 的核心——你在這裡用拖拉的方式設計整個自動化流程。

### 建立新流程

1. 在流程清單頁面，點選右上角的**新增流程**按鈕
2. 輸入流程名稱和說明
3. 系統會開啟視覺化流程編輯器

<!-- TODO: 截圖 — 新增流程對話框 -->

### 編輯器操作

1. 畫面左側是**節點面板**，列出所有可用的節點類型（觸發器、HTTP 呼叫、條件判斷等）
2. 將節點從面板**拖拉到畫布**上
3. 將滑鼠移到節點的連接點，按住拖曳一條**連線**到下一個節點，建立執行順序
4. **點擊節點**開啟右側的參數設定面板，設定該節點的詳細參數
5. 重複步驟 2-4，直到完成整個流程的設計

<!-- TODO: 截圖 — 視覺化編輯器畫布，含數個已連接的節點 -->

### 即時驗證

編輯器會**即時檢查**你的流程設計是否正確：

- 缺少必填參數的節點會顯示紅色警示圖示
- 連線錯誤（例如沒有連接的孤立節點）會以虛線標示
- 儲存或發布前，系統會列出所有需要修正的問題

> **提示：** 建議每加入幾個節點就檢查一次驗證訊息，不要等到整個流程都設計完才回頭修正。

## 版本控制

**適用角色：** IT 管理員

每個工作流程都有獨立的版本歷史。你可以放心修改流程設計，因為每次發布都會建立新版本，隨時可以回到舊版。

### 發布新版本

1. 在流程編輯器中完成修改
2. 點選右上角的**發布**按鈕
3. 系統建立新版本（例如 v1 → v2）並立即生效

> **提示：** 未發布的編輯不會影響正在執行的版本。你可以放心在編輯器中調整，等到確認沒問題再發布。

### 查看版本歷史

1. 在流程清單中，點選目標流程進入詳情頁
2. 切換到**版本歷史**標籤
3. 你會看到所有已發布的版本清單，每筆顯示版本號、發布時間和發布者

<!-- TODO: 截圖 — 版本歷史清單 -->

### 比較版本差異

1. 在版本歷史清單中，勾選**兩個版本**
2. 點選**比較差異**按鈕
3. 系統會並排顯示兩個版本的流程結構差異，標示新增、刪除和修改的節點

### 回滾到舊版本

1. 在版本歷史清單中，找到要恢復的版本
2. 點選該版本的**重新啟用**按鈕
3. 確認後，系統會將該版本設為目前使用版本

> **注意：** 回滾會建立一個新版本（內容和選定的舊版本相同），不會刪除中間的版本紀錄。

## 流程節點類型

Automation Platform 提供多種節點類型，你可以自由組合來建構各種自動化流程。以下逐一說明每種節點的用途、設定方式，並附上 IT 資安維運的實際範例。

### HTTP 呼叫

呼叫外部 REST API，讓流程與其他系統互動。

#### 可設定參數

| 參數 | 說明 |
|------|------|
| URL | 目標 API 的端點網址，支援變數帶入（例如 `https://ad.example.com/api/users/{{user_id}}`） |
| Method | HTTP 方法：GET / POST / PUT / PATCH / DELETE |
| Headers | 自訂 HTTP 標頭 |
| Body | 請求內容，支援 JSON 格式，可用變數帶入動態值 |
| Auth | 認證方式：OAuth 2.0 / API Key / Basic Auth |
| 錯誤處理 | 失敗時的處理邏輯：重試 / 跳過 / 中止流程 |

#### 運作方式

- 設定好的 Request 會在流程執行到這個節點時送出
- API Response 會自動解析，結果可以傳給下一個節點使用
- 如果 API 回傳錯誤，依「錯誤處理」設定決定下一步動作

#### 實際範例：自動停用可疑帳號

**場景：** 資安系統偵測到某個 AD 帳號有異常登入行為，流程自動呼叫 AD API 停用該帳號。

| 參數 | 設定值 |
|------|--------|
| URL | `https://ad.corp.local/api/v1/users/{{suspicious_user}}/disable` |
| Method | POST |
| Auth | OAuth 2.0（選擇已設定的 AD 連線） |
| Body | `{"reason": "{{alert_description}}", "disabled_by": "automation"}` |
| 錯誤處理 | 重試 2 次，間隔 5 秒；仍失敗則中止流程 |

### 腳本執行

在隔離沙盒中執行自訂的 Python 或 Shell Script。

#### 可設定參數

| 參數 | 說明 |
|------|------|
| 語言 | Python 或 Shell Script |
| 腳本內容 | 直接在編輯器中撰寫腳本 |
| 逾時時間 | 腳本執行的最長時間，超過自動終止 |

#### 運作方式

- 腳本在**隔離沙盒**中執行，不會影響主系統
- 腳本的標準輸出（stdout）會傳給下一個節點作為輸入
- 可以在腳本中使用前面節點傳入的變數

#### 實際範例：分析防火牆日誌中的異常 IP

**場景：** 定期從防火牆取得日誌，用 Python 腳本分析是否有同一 IP 大量嘗試連線的行為。

```python
import json

# 從前一個 HTTP 呼叫節點取得防火牆日誌
logs = json.loads(input_data)

# 統計每個 IP 的連線次數
ip_counts = {}
for entry in logs:
    ip = entry.get("source_ip")
    ip_counts[ip] = ip_counts.get(ip, 0) + 1

# 找出超過 100 次的可疑 IP
suspicious = {ip: count for ip, count in ip_counts.items() if count > 100}

# 輸出給下一個節點
print(json.dumps({"suspicious_ips": suspicious, "total_checked": len(logs)}))
```

| 參數 | 設定值 |
|------|--------|
| 語言 | Python |
| 逾時時間 | 30 秒 |

### 條件判斷

依據變數值決定流程走哪個分支。

#### 可設定參數

| 參數 | 說明 |
|------|------|
| 變數 | 要比對的變數名稱（來自前面節點的輸出） |
| 運算子 | 等於 / 不等於 / 包含 / 大於 / 小於 |
| 比對值 | 要比對的目標值 |
| 條件組合 | 多個條件之間用 AND 或 OR 連接 |

#### 運作方式

- 條件判斷節點有兩個出口：**符合**和**不符合**
- 你可以設定多個條件，用 AND（全部符合）或 OR（任一符合）組合
- 符合條件走一條分支，不符合走另一條分支

#### 實際範例：依事件嚴重度決定處理流程

**場景：** 資安事件進入後，依嚴重度判斷：SEV1/SEV2 立即通知資安主管並啟動緊急應變流程，SEV3/SEV4 進入一般處理佇列。

| 參數 | 設定值 |
|------|--------|
| 變數 | `incident.severity` |
| 運算子 | 等於 |
| 比對值 | `SEV1` |
| 條件組合 | OR `incident.severity` 等於 `SEV2` |

- **符合分支** → 通知資安主管 + 啟動緊急應變 Runbook
- **不符合分支** → 建立一般工單，指派給值班人員

### 人工簽核

在流程中加入人工審核關卡，流程會暫停等待指定人員核准或退回。

#### 可設定參數

| 參數 | 說明 |
|------|------|
| 簽核流程 | 選用哪個簽核流程定義（來自核心簽核引擎） |
| 簽核人覆寫 | 選填，可覆寫簽核流程中預設的簽核人 |

#### 運作方式

- 流程執行到簽核節點時會**暫停**，等待簽核結果
- 簽核人會收到通知（Email / 站內通知），並可在通知中直接核准或退回
- **核准**：流程繼續執行下一個節點
- **退回**：流程走失敗分支

> 簽核節點呼叫的是 Arova Nexus 核心簽核引擎，支援多級串簽、逾時自動提醒等進階功能。簽核引擎的詳細設定請參考 [管理員設定指南](06-admin-setup.md)。

#### 實際範例：高權限帳號變更需雙重簽核

**場景：** 有人申請開通 Domain Admin 權限，流程自動要求資安主管和 IT 主管雙重核准。

| 參數 | 設定值 |
|------|--------|
| 簽核流程 | 「高權限變更審核」（多級串簽：資安主管 → IT 主管） |
| 簽核人覆寫 | 不覆寫（使用簽核流程預設的簽核人） |

流程結構：申請提交 → **人工簽核（雙重核准）** → HTTP 呼叫 AD 開通權限 → 通知申請人

### 通知

在流程中發送通知給相關人員。

#### 可設定參數

| 參數 | 說明 |
|------|------|
| 通知通道 | Email / Slack / Webhook |
| 通知範本 | 選用預設範本或自訂內容 |
| 變數 | 範本中的動態內容，從前面節點的輸出帶入 |

#### 運作方式

- 支援同時發送到多個通道
- 範本支援變數插入，例如 `{{user_name}}`、`{{incident_id}}`
- 通知發送後流程繼續執行，不會暫停

#### 實際範例：資安事件升級通知

**場景：** 資安事件自動升級時，同時透過 Email 和 Slack 通知資安團隊全員。

| 參數 | 設定值 |
|------|--------|
| 通知通道 | Email + Slack |
| 收件人（Email） | 資安團隊群組 `security-team@corp.local` |
| Slack 頻道 | `#security-alerts` |
| 範本 | 「資安事件升級通知」 |
| 變數 | `{{incident_id}}`、`{{severity}}`、`{{description}}`、`{{source_ip}}` |

範本內容預覽：
```
[緊急] 資安事件 {{incident_id}} 已升級為 {{severity}}
事件描述：{{description}}
來源 IP：{{source_ip}}
請立即至 Arova Nexus 查看處理。
```

### 工單操作

在流程中自動操作工單，例如更新狀態、指派處理人或加入留言。

> 📋 **需要的模組：** 此節點需要 Service Desk 模組。如果你的系統未啟用 Service Desk，此節點會在編輯器中顯示為灰色鎖定狀態。若需啟用，請聯繫管理員了解模組升級方案。

#### 可設定參數

| 參數 | 說明 |
|------|------|
| 目標工單 | 指定要操作的工單（可用變數帶入工單編號） |
| 動作 | 更新狀態 / 指派處理人 / 加入留言 |
| 狀態值 | 要更新的目標狀態 |
| 處理人 | 要指派的人員 |
| 留言內容 | 要加入的留言，支援變數 |

#### 實際範例：資安事件自動建立工單並指派

**場景：** 防火牆偵測到入侵行為，流程自動在 Service Desk 建立高優先工單，指派給值班的資安工程師，並附上分析結果。

| 參數 | 設定值 |
|------|--------|
| 目標工單 | `{{ticket_id}}`（由前面的「建立工單」節點產生） |
| 動作 | 指派處理人 + 加入留言 |
| 處理人 | `{{on_duty_engineer}}`（值班人員） |
| 留言內容 | `自動分析結果：來源 IP {{source_ip}} 在過去 1 小時內嘗試 {{attempt_count}} 次未授權存取。建議立即封鎖。` |

### 等待與迴圈

控制流程的時間節奏和重複執行邏輯。

#### 等待（Delay）

| 參數 | 說明 |
|------|------|
| 等待模式 | 固定時間 / 等到條件成立 |
| 固定時間 | 等待的時間長度（秒 / 分 / 時） |
| 條件 | 等到某個變數符合指定條件時繼續 |

#### 迴圈（Loop）

| 參數 | 說明 |
|------|------|
| 迭代清單 | 要逐一處理的清單（來自前面節點的輸出） |
| 最大次數 | 迴圈執行的上限次數 |
| 中止條件 | 符合此條件時提前結束迴圈 |

#### 實際範例：批次掃描多台伺服器的弱點修補狀態

**場景：** 每月定期檢查所有關鍵伺服器是否已安裝最新安全性更新，對清單中的每台伺服器逐一呼叫 API 檢查。

**迴圈節點設定：**

| 參數 | 設定值 |
|------|--------|
| 迭代清單 | `{{server_list}}`（從 CMDB 取得的伺服器清單） |
| 最大次數 | 200 |
| 中止條件 | 無（全部跑完） |

迴圈內的子流程：HTTP 呼叫檢查更新 → 條件判斷（是否已修補） → 未修補則通知負責人

**等待節點設定（另一個範例）：**

| 參數 | 設定值 |
|------|--------|
| 等待模式 | 固定時間 |
| 固定時間 | 300 秒（5 分鐘） |

> **提示：** 等待節點的「等到條件成立」模式很適合用在需要等待外部系統完成處理的場景，例如等 AD 同步完成後再執行下一步。

### AI 呼叫

在流程中呼叫 AI 模型，用自然語言處理資料、做分析或產生建議。

#### 可設定參數

| 參數 | 說明 |
|------|------|
| Prompt 範本 | 選用預設範本或自訂 Prompt |
| 輸入變數 | 從前面節點傳入的資料，作為 Prompt 的動態內容 |
| 輸出格式 | 指定 AI 回應的格式（JSON Schema），讓後續節點可以結構化使用 |

#### 運作方式

- AI 呼叫節點會把你的 Prompt 和輸入變數組合後送給 AI 模型
- AI 的回應會依照你指定的輸出格式（JSON Schema）結構化
- 結構化的輸出可以傳給下一個節點，作為條件判斷或其他動作的依據

#### 實際範例：AI 自動分類資安事件並建議處置方式

**場景：** 資安警報進來後，AI 自動分析事件描述，判斷攻擊類型並建議初步處置步驟。

| 參數 | 設定值 |
|------|--------|
| Prompt 範本 | 自訂 |
| 輸入變數 | `{{alert_description}}`、`{{source_ip}}`、`{{target_system}}` |
| 輸出格式（JSON Schema） | `{"attack_type": "string", "risk_level": "high/medium/low", "recommended_actions": ["string"]}` |

Prompt 內容：
```
你是一位資安分析師。根據以下警報資訊，判斷攻擊類型、風險等級，並建議處置步驟。

警報描述：{{alert_description}}
來源 IP：{{source_ip}}
目標系統：{{target_system}}

請依照指定的 JSON 格式輸出。
```

AI 回應後，下一個條件判斷節點可以根據 `risk_level` 決定是否啟動緊急應變流程。

## 即時監控

**適用角色：** IT 維運人員

流程開始執行後，你可以即時追蹤每個節點的狀態。

### 操作步驟

1. 在流程清單中，點選正在執行的流程，或從執行紀錄中點選某筆執行
2. 系統會顯示流程的**圖形化執行視圖**
3. 每個節點會用顏色和圖示標示目前狀態：
   - ⚪ 灰色 = 待執行
   - 🔵 藍色（帶動畫）= 正在執行
   - 🟢 綠色 = 已完成
   - 🔴 紅色 = 失敗
   - ⏭️ 斜線 = 已跳過
4. 頁面上方會顯示**整體進度百分比**
5. **點擊任何節點**可以查看該節點的輸入資料和輸出結果

## 失敗處理與重試

**適用角色：** IT 維運人員

當流程中的某個節點執行失敗時，系統提供自動重試和手動重試兩種機制。

### 自動重試

- 每個節點可以在設計時設定**自動重試次數**和**重試間隔**
- 節點失敗後，系統會自動按照設定的間隔重試
- 如果所有重試都失敗，流程標記為「失敗」

### 手動重試

1. 在執行歷史中找到失敗的執行紀錄
2. 點選進入執行詳情頁
3. 找到標示為紅色的失敗節點
4. 點選該節點的**重試**按鈕
5. 流程會**從該節點重新開始執行**，已完成的節點不會重跑

<!-- TODO: 截圖 — 失敗節點的重試按鈕 -->

> **提示：** 手動重試前，建議先點擊失敗節點查看錯誤訊息，確認問題已排除（例如外部 API 已恢復、網路已通）再重試。

> **注意：** 所有重試的紀錄都會完整保留在執行歷史中，包含每次重試的時間、輸入、輸出和結果。

## 強制終止

**適用角色：** IT 維運人員

如果流程執行異常或不再需要，你可以強制終止正在執行的流程。

### 操作步驟

1. 在即時監控視圖或執行歷史中，找到正在執行的流程
2. 點選**強制終止**按鈕
3. 系統會彈出確認對話框，要求你輸入終止原因
4. 輸入原因後，點選**確認終止**
5. 流程立即停止：
   - 已完成的節點結果**不受影響**
   - 未完成的節點標記為「已終止」
6. 終止原因和操作人會記錄在執行歷史中

## 外部觸發

**適用角色：** IT 管理員

你可以讓外部系統透過 Webhook 或 REST API 自動觸發工作流程，實現跨系統的自動化串接。

### Webhook 觸發

1. 在流程清單中，點選目標流程進入詳情頁
2. 切換到**觸發設定**標籤
3. 點選**產生 Webhook URL**
4. 系統會產生一個專屬的 Webhook URL，例如：
   ```
   https://nexus.corp.local/api/webhooks/wf/abc123-def456
   ```
5. 將此 URL 設定到外部系統（例如 Graylog 的 Alert Notification、LibreNMS 的 Alert Transport）
6. 外部系統對此 URL 發送 `POST` 請求即可觸發流程
7. POST 請求的 Payload 資料會自動成為流程的輸入參數

### API 觸發

除了 Webhook，你也可以使用 REST API 搭配 API Key 認證來觸發流程：

```
POST /api/v1/workflows/{workflow_id}/execute
Authorization: Bearer {api_key}
Content-Type: application/json

{
  "parameters": {
    "param1": "value1",
    "param2": "value2"
  }
}
```

> **提示：** API Key 可以在**設定** → **API Key 管理**中產生。建議為每個外部系統建立獨立的 API Key，方便管理和撤銷。

### 觸發紀錄

所有外部觸發都會完整記錄，你可以在**執行歷史**中看到觸發方式標示為「Webhook」或「API」，並查看觸發時傳入的 Payload 內容。

## 排程管理

**適用角色：** IT 管理員

你可以為工作流程設定排程，讓流程在指定的時間自動執行。

### 設定排程

1. 在流程清單中，點選目標流程進入詳情頁
2. 切換到**排程**標籤
3. 點選**新增排程**
4. 輸入 Cron 表達式，或使用介面的輔助工具選擇時間
5. 介面會自動顯示**人話翻譯**，例如：
   - `0 2 * * *` → 「每天凌晨 2:00」
   - `0 9 * * 1` → 「每週一早上 9:00」
   - `0 0 1 * *` → 「每月 1 日凌晨 0:00」
6. 確認無誤後，點選**儲存**，排程立即生效

## Dry-Run 測試

**適用角色：** IT 管理員

在正式發布流程之前，使用 Dry-Run 模式進行模擬測試，確認流程邏輯是否正確。

### 操作步驟

1. 在流程編輯器中，完成流程設計後點選**Dry-Run 測試**按鈕
2. 如果流程有輸入參數，填寫測試用的參數值
3. 點選**開始測試**
4. 系統會**模擬執行**所有節點：
   - HTTP 呼叫節點不會實際送出 API 請求，而是模擬回應
   - 通知節點不會實際發送 Email 或 Slack 訊息
   - 腳本節點會在沙盒中執行，但不會與外部系統互動
5. 每個節點會顯示模擬的輸入和輸出結果
6. 測試完成後，系統顯示**整體結果摘要**，列出每個節點的模擬狀態

## 內建流程範本

**適用角色：** IT 管理員

Automation Platform 內建五個常見 IT 維運場景的流程範本，讓你不用從零開始設計。

### 使用範本

1. 在流程清單頁面，點選**新增流程**
2. 在對話框中選擇**從範本建立**
3. 瀏覽可用的範本清單
4. 選擇範本後，點選**複製為新流程**
5. 系統會建立一個該範本的副本，你可以自由修改節點、參數和流程邏輯
6. 修改完成後，按照正常流程進行 Dry-Run 測試並發布

### 可用範本

#### 密碼重置

| 項目 | 說明 |
|------|------|
| 流程 | 使用者提交申請 → 驗證身份 → 呼叫 AD API 重置密碼 → 通知使用者新密碼 |
| 整合系統 | AD / LDAP |
| 適用場景 | 使用者忘記密碼，自助或由 IT 人員協助重置 |

#### VPN 申請

| 項目 | 說明 |
|------|------|
| 流程 | 使用者提交申請 → 多級簽核（主管 → IT 主管） → 自動開通 VPN 帳號 → 通知申請人 |
| 整合系統 | VPN Gateway |
| 適用場景 | 員工申請遠端 VPN 存取權限 |

#### 軟體安裝

| 項目 | 說明 |
|------|------|
| 流程 | 使用者提交申請 → 檢查軟體授權數量 → 自動部署安裝 → 安裝確認 |
| 整合系統 | SCCM / Intune |
| 適用場景 | 員工申請安裝特定軟體 |

#### 帳號解鎖

| 項目 | 說明 |
|------|------|
| 流程 | 使用者提交申請 → 驗證身份 → 呼叫 AD API 解鎖帳號 → 通知使用者 |
| 整合系統 | AD / LDAP |
| 適用場景 | 使用者因多次輸入錯誤密碼導致帳號被鎖定 |

#### 新人到職

| 項目 | 說明 |
|------|------|
| 流程 | HR 系統觸發 → 自動建立 AD 帳號 → 開通各系統權限 → 配發設備 → 寄送歡迎通知 |
| 整合系統 | AD / HR 系統 / 資產庫 |
| 適用場景 | 新員工到職時，一次完成所有 IT 環境的準備工作 |

> **提示：** 範本提供的是通用流程，你需要根據自己組織的系統環境和流程規範進行調整。例如「密碼重置」範本中的 AD API 端點需要改成你們實際的 AD 伺服器位址。

## 常見問題

**Q：流程正在執行中，我可以修改流程設計嗎？**
A：可以。你在編輯器中的修改不會影響正在執行的版本。修改完成後點選**發布**才會建立新版本，新版本只會套用到之後的執行。

**Q：排程觸發的流程失敗了，排程會停止嗎？**
A：不會。排程會持續按照設定的時間觸發。但如果你發現連續多次失敗，建議檢查流程設計和外部系統的連線狀態。

**Q：Webhook URL 可以重新產生嗎？**
A：可以。重新產生後舊的 URL 會立即失效，記得同步更新所有使用舊 URL 的外部系統設定。

**Q：Dry-Run 和正式執行的結果會不同嗎？**
A：有可能。Dry-Run 不會實際呼叫外部 API，所以如果你的流程邏輯依賴外部 API 的真實回應，Dry-Run 的模擬結果可能與正式執行不同。Dry-Run 主要用來驗證流程結構和條件判斷的邏輯。

**Q：工單操作節點是灰色的，無法使用？**
A：工單操作節點需要 Service Desk 模組。如果你的系統未啟用 Service Desk，此節點會顯示為灰色鎖定狀態。請聯繫管理員了解模組升級方案。
